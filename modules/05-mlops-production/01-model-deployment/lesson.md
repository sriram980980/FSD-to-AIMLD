# 5.1 — Inference Serving — High-Throughput Engines

## Hook

Continuous batching in vLLM is like connection pooling — instead of allocating one thread per request and blocking it until the response is complete, vLLM dynamically groups in-flight generation requests to maximize GPU utilization, exactly as a database pool maximizes connection reuse across concurrent queries.

## The Problem

A naive inference server processes one prompt at a time: load the model, run the forward pass, return the output, repeat. At production scale this is catastrophically slow — a single H100 GPU can sustain tens of thousands of tokens/second yet a sequential server delivers only hundreds, leaving ~95 % of compute idle. Worse, LLMs generate tokens one at a time, so long-running requests block short ones, spiking P99 latency and wasting expensive GPU-hours. High-throughput inference engines — vLLM, TensorRT-LLM — exist to close this utilization gap.

## Theory

### Continuous Batching and KV-Cache Paging

**The core bottleneck: GPU memory fragmentation**

Classic static batching reserves a fixed-size KV (key-value) cache slot per request at arrival. If request A needs 4 096 tokens and request B needs 16 tokens, A holds 256× more memory — but you must reserve the worst case upfront. Most slots sit empty, limiting batch size.

**PagedAttention** (the technique inside vLLM) borrows virtual memory paging from OS design:

$$\text{KV cache} = \bigcup_{i=1}^{B} \text{blocks}(r_i), \quad |\text{block}| = k \text{ tokens}$$

- $B$ — current batch size (dynamic, changes every decode step)
- $r_i$ — request $i$ currently in flight
- $k$ — block size, e.g. 16 tokens (a tunable constant)
- Each block is a contiguous physical memory page; logical pages are mapped to physical pages via a block table (just like OS virtual memory)

**Throughput formula:**

$$\text{Throughput (tok/s)} = \frac{\sum_{i=1}^{B} L_i}{T_{\text{batch}}}$$

- $L_i$ — tokens generated by request $i$ in this decode step (always 1 token per step per request)
- $T_{\text{batch}}$ — wall-clock time for one decode step
- Since $L_i = 1$ for all $i$: Throughput $= B / T_{\text{batch}}$

**Numeric worked example:**

A static server handles $B = 4$ requests (GPU memory fully allocated). One decode step takes $T_{\text{batch}} = 20\,\text{ms}$:

$$\text{Throughput}_{\text{static}} = \frac{4}{0.020} = 200\,\text{tok/s}$$

With PagedAttention, freed pages from completed requests are immediately reallocated to new arrivals. The effective batch grows to $B = 32$:

$$\text{Throughput}_{\text{paged}} = \frac{32}{0.020} = 1{,}600\,\text{tok/s}$$

An **8×** improvement on the same hardware — no model change, no quantization.

### Quantization: INT8 and INT4

Weights are stored as FP16 (2 bytes/param). INT8 quantization halves that:

$$W_{\text{int8}} = \text{round}\!\left(\frac{W_{\text{fp16}}}{s}\right), \quad s = \frac{\max(|W_{\text{fp16}}|)}{127}$$

- $W_{\text{fp16}}$ — original weight value
- $s$ — per-tensor scale factor (stored as FP32)
- $W_{\text{int8}}$ — quantized integer weight in $[-127, 127]$

Worked example — single weight $w = 0.48$, tensor max $= 0.96$:

$$s = \frac{0.96}{127} \approx 0.00756, \quad W_{\text{int8}} = \text{round}\!\left(\frac{0.48}{0.00756}\right) = \text{round}(63.5) = 64$$

Recovery: $\hat{w} = 64 \times 0.00756 = 0.4838$ — only $0.79\,\%$ error for this weight.

A 7 B-parameter model at FP16 occupies $7 \times 10^9 \times 2 = 14\,\text{GB}$ of VRAM. INT8 cuts it to $7\,\text{GB}$; INT4 (GPTQ/AWQ) cuts it to $3.5\,\text{GB}$, fitting a consumer GPU.

### TensorRT-LLM: Compiled Inference Kernels

vLLM is Python-managed, dispatching PyTorch CUDA kernels at runtime. TensorRT-LLM compiles the full inference graph into optimized CUDA kernels at deploy time — equivalent to AOT compilation vs. JIT. The payoff: **20–40 % lower latency** at the same throughput, at the cost of a longer build step (5–30 min per model/GPU pair). Use vLLM for flexibility; use TensorRT-LLM when you have a fixed model on a fixed GPU SKU and need every millisecond.

## Python Implementation

```python
# Dependencies: vllm>=0.4, fastapi>=0.111, uvicorn>=0.29
"""
Lesson 5.1 — Inference Serving with vLLM + FastAPI

Demonstrates:
  1. Offline batch throughput measurement (no GPU? falls back to mock).
  2. A minimal FastAPI server wrapping vLLM's AsyncLLMEngine.
  3. Continuous-batching behaviour made observable via request timing.

Run (requires a GPU with >=8 GB VRAM):
  pip install vllm>=0.4 fastapi>=0.111 uvicorn>=0.29
  python lesson.py --mode benchmark
  python lesson.py --mode server   # then: curl http://localhost:8000/generate -d '{"prompt":"Hello","max_tokens":50}'
"""

import argparse
import asyncio
import time
from typing import AsyncIterator

# ── Attempt real vLLM import; fall back to a mock for CPU-only environments ──
try:
    from vllm import LLM, SamplingParams
    from vllm.engine.arg_utils import AsyncEngineArgs
    from vllm.engine.async_llm_engine import AsyncLLMEngine
    VLLM_AVAILABLE = True
except ImportError:
    VLLM_AVAILABLE = False
    print("INFO: vLLM not installed or no CUDA GPU detected — running in MOCK mode.")

import fastapi
import uvicorn


# ─────────────────────────────────────────────
# 1.  Throughput benchmark (offline, batched)
# ─────────────────────────────────────────────

MODEL_ID = "facebook/opt-125m"   # tiny model, fits on any GPU with >=1 GB VRAM


def run_benchmark(num_prompts: int = 32, max_tokens: int = 64) -> None:
    """Measure tokens/second under static vs. continuous batching."""
    prompts = [f"Explain concept number {i} in machine learning:" for i in range(num_prompts)]
    sampling_params = SamplingParams(temperature=0.0, max_tokens=max_tokens)

    if not VLLM_AVAILABLE:
        _mock_benchmark(prompts, max_tokens)
        return

    print(f"\n── Benchmark: {num_prompts} prompts × {max_tokens} tokens ──")
    llm = LLM(model=MODEL_ID, dtype="float16")

    t0 = time.perf_counter()
    outputs = llm.generate(prompts, sampling_params)
    elapsed = time.perf_counter() - t0

    total_tokens = sum(len(o.outputs[0].token_ids) for o in outputs)
    throughput = total_tokens / elapsed

    print(f"Total tokens generated : {total_tokens}")
    print(f"Wall-clock time        : {elapsed:.2f} s")
    print(f"Throughput             : {throughput:.1f} tok/s")
    print(f"Effective batch size   : {num_prompts} requests processed concurrently")


def _mock_benchmark(prompts: list[str], max_tokens: int) -> None:
    """CPU-safe mock that illustrates the throughput calculation."""
    import random
    random.seed(42)
    num_prompts = len(prompts)
    print(f"\n── MOCK Benchmark: {num_prompts} prompts × {max_tokens} tokens ──")

    # Simulate static batching (batch=4)
    static_batch = 4
    static_step_ms = 20.0  # ms per decode step
    static_steps = (num_prompts + static_batch - 1) // static_batch
    static_total_s = static_steps * static_step_ms / 1000
    static_tokens = num_prompts * max_tokens
    static_throughput = static_tokens / static_total_s

    # Simulate continuous batching (effective batch grows to ~num_prompts)
    cont_batch = num_prompts
    cont_total_s = (num_prompts * max_tokens) / cont_batch * (static_step_ms / 1000)
    cont_throughput = (cont_batch * max_tokens) / cont_total_s

    print(f"Static batching  (batch={static_batch}) → {static_throughput:.1f} tok/s")
    print(f"Continuous batch (batch≈{cont_batch})  → {cont_throughput:.1f} tok/s")
    print(f"Speedup          : {cont_throughput / static_throughput:.1f}×")


# ─────────────────────────────────────────────
# 2.  FastAPI + AsyncLLMEngine server
# ─────────────────────────────────────────────

app = fastapi.FastAPI(title="vLLM Inference Server — Lesson 5.1")
engine: "AsyncLLMEngine | None" = None  # initialised on startup


@app.on_event("startup")
async def startup() -> None:
    global engine
    if not VLLM_AVAILABLE:
        print("INFO: Server running in MOCK mode (no real generation).")
        return
    args = AsyncEngineArgs(model=MODEL_ID, dtype="float16")
    engine = AsyncLLMEngine.from_engine_args(args)


class GenerateRequest(fastapi.BaseModel):
    prompt: str
    max_tokens: int = 64
    temperature: float = 0.7


class GenerateResponse(fastapi.BaseModel):
    text: str
    tokens_generated: int
    latency_ms: float


@app.post("/generate", response_model=GenerateResponse)
async def generate(request: GenerateRequest) -> GenerateResponse:
    """Single-turn generation endpoint. vLLM handles batching internally."""
    t0 = time.perf_counter()

    if engine is None:
        # MOCK path
        mock_text = f"[MOCK] Response to: {request.prompt[:40]}..."
        return GenerateResponse(
            text=mock_text,
            tokens_generated=request.max_tokens,
            latency_ms=(time.perf_counter() - t0) * 1000,
        )

    sampling_params = SamplingParams(
        temperature=request.temperature,
        max_tokens=request.max_tokens,
    )
    import uuid
    request_id = str(uuid.uuid4())

    # Stream tokens until generation completes
    final_output = None
    async for request_output in engine.generate(request.prompt, sampling_params, request_id):
        final_output = request_output

    latency_ms = (time.perf_counter() - t0) * 1000
    generated_text = final_output.outputs[0].text
    tokens_generated = len(final_output.outputs[0].token_ids)

    print(f"Request done: {tokens_generated} tokens in {latency_ms:.1f} ms "
          f"({tokens_generated / (latency_ms / 1000):.1f} tok/s)")

    return GenerateResponse(
        text=generated_text,
        tokens_generated=tokens_generated,
        latency_ms=latency_ms,
    )


@app.get("/health")
async def health() -> dict[str, str]:
    return {"status": "ok", "backend": "vllm" if VLLM_AVAILABLE else "mock"}


# ─────────────────────────────────────────────
# 3.  Quantization memory math demo (CPU-safe)
# ─────────────────────────────────────────────

def quantization_demo() -> None:
    """Show INT8 quantization arithmetic — no GPU required."""
    import numpy as np

    print("\n── INT8 Quantization Demo ──")
    rng = np.random.default_rng(0)
    weights_fp16 = rng.uniform(-1.0, 1.0, size=(4, 4)).astype(np.float32)

    w_max = np.max(np.abs(weights_fp16))
    scale = w_max / 127.0
    weights_int8 = np.round(weights_fp16 / scale).astype(np.int8)
    weights_recovered = weights_int8.astype(np.float32) * scale

    max_error = np.max(np.abs(weights_fp16 - weights_recovered))
    mean_error = np.mean(np.abs(weights_fp16 - weights_recovered))

    print(f"Original FP32 weights (4×4):\n{np.round(weights_fp16, 4)}")
    print(f"\nScale factor  : {scale:.6f}  (max_weight / 127)")
    print(f"INT8 weights  :\n{weights_int8}")
    print(f"\nRecovered FP32 (INT8 → FP32):\n{np.round(weights_recovered, 4)}")
    print(f"\nMax abs error : {max_error:.6f}")
    print(f"Mean abs error: {mean_error:.6f}")

    param_count = 7_000_000_000  # 7B model
    fp16_gb = param_count * 2 / 1e9
    int8_gb = param_count * 1 / 1e9
    int4_gb = param_count * 0.5 / 1e9
    print(f"\n── 7B Model VRAM footprint ──")
    print(f"FP16 : {fp16_gb:.1f} GB")
    print(f"INT8 : {int8_gb:.1f} GB  (2× compression)")
    print(f"INT4 : {int4_gb:.1f} GB  (4× compression — fits consumer GPU)")


# ─────────────────────────────────────────────
# Entry point
# ─────────────────────────────────────────────

def main() -> None:
    parser = argparse.ArgumentParser(description="Lesson 5.1 — Inference Serving")
    parser.add_argument(
        "--mode",
        choices=["benchmark", "server", "quantize"],
        default="quantize",
        help="benchmark: throughput test | server: run API | quantize: INT8 math demo",
    )
    args = parser.parse_args()

    if args.mode == "benchmark":
        run_benchmark(num_prompts=32, max_tokens=64)
    elif args.mode == "server":
        print("Starting vLLM FastAPI server on http://localhost:8000")
        uvicorn.run(app, host="0.0.0.0", port=8000, log_level="info")
    else:
        quantization_demo()


if __name__ == "__main__":
    main()
```

Run `python lesson.py` (no arguments) to see the INT8 quantization arithmetic — no GPU required. Notice:

- The **scale factor** maps the full FP16 range to $[-127, 127]$; the max absolute error on individual weights is typically below $0.005$ — negligible for inference quality.
- The 7 B model VRAM table makes the motivation visceral: INT4 drops you from a $14\,\text{GB}$ requirement to $3.5\,\text{GB}$, unlocking deployment on a single RTX 4090.
- Run `--mode benchmark` on a GPU machine to observe actual continuous-batching throughput; on CPU it runs the mock that shows the arithmetic behind the speedup calculation.
- Run `--mode server` to start the FastAPI endpoint; hit `GET /health` to confirm the backend and `POST /generate` to generate text.

## Key Takeaways

- **Continuous batching + PagedAttention** eliminate static KV-cache waste: completed requests immediately free memory pages for new arrivals, multiplying effective batch size (and throughput) by 4–10× on the same GPU hardware.
- **Quantization (INT8/INT4)** compresses model weights with sub-1 % per-weight error, halving or quartering VRAM requirements and enabling 7 B+ models to run on consumer GPUs — pick the precision tier that fits your accuracy/cost target.
- **vLLM vs. TensorRT-LLM** is a flexibility-vs-latency trade-off: vLLM is model-agnostic and fast to deploy; TensorRT-LLM AOT-compiles inference graphs to squeeze 20–40 % more performance from a fixed model-GPU pair.
